================================================================
  SECTION 4 — NORMALIZATION SUMMARY
================================================================

  1NF (First Normal Form) — Atomic values, no repeating groups
  ─────────────────────────────────────────────────────────────
  Status:  ✓ Was already correct. All cells hold single values.
  Example: users.role holds ONE value ('investor', not 'investor,admin')


  2NF (Second Normal Form) — Full dependency on the primary key
  ──────────────────────────────────────────────────────────────
  Status:  ✓ Was already correct. All PKs are single UUIDs.
           With a single-column PK, partial dependency is impossible.


  3NF (Third Normal Form) — No transitive dependencies
  ───────────────────────────────────────────────────────
  Status:  ✗ Had a violation → ✓ Fixed.

  PROBLEM FOUND:
    projects had columns: category, category_ar, category_en
    category_ar and category_en depend on "category", not on the project "id"
    This means the project's PK → category → category_ar (transitive)

  FIX APPLIED:
    Created a "categories" table:
      id (PK) | name_ar | name_en | icon
    Removed category_ar, category_en from projects table.
    Added FOREIGN KEY: projects.category → categories.id

  BONUS FIX:
    notifications.is_read was a single flag on a row shared by all users
    (broadcast notifications). This couldn't correctly represent each
    user's individual read state.

    Fix: Created user_notification_reads join table.
    Each (user_id, notification_id) pair gets its own is_read column.


================================================================
  SECTION 5 — JOIN TABLES
================================================================

  A join table (junction table) sits between two tables in a M:M
  relationship. It has foreign keys to both sides and may carry
  its own columns (extra data about the relationship).

  Join Table 1: investments
  ─────────────────────────
  Connects:  users (as investors) ↔ projects
  Columns:   investor_id (FK), project_id (FK),
             amount, currency_code, payment_method, status, reference

  Meaning: "Ahmed invested 2500 LYD in Tech Platform via wallet, completed"
  This data belongs to the RELATIONSHIP, not to Ahmed and not to the project.

  Join Table 2: user_notification_reads
  ──────────────────────────────────────
  Connects:  users ↔ notifications
  Columns:   user_id (FK), notification_id (FK), is_read, read_at

  Meaning: "Sara has NOT read the Welcome broadcast notification"
  The read state belongs to the USER-NOTIFICATION pair, not to either alone.


================================================================
  SECTION 6 — VIEWS (pre-written JOIN queries)
================================================================

  Instead of writing JOIN queries in the backend code every time,
  we wrote them once as views. The backend just does SELECT * FROM view.

  View                       Tables Joined
  ─────────────────────────  ──────────────────────────────────────────
  vw_project_summary         projects + categories + users (owner)
                             → gives: category names, owner name, progress %

  vw_user_summary            users + wallets + investments + projects
                             → gives: balance, total invested, project count

  vw_investment_details      investments + projects + users (investor)
                             → gives: project title, investor name, amount

  vw_payment_details         payments + users (payer) + users (approver)
                             → uses a SELF-JOIN on users twice

  vw_wallet_details          wallets + users
                             → gives: wallet balance with user info

  vw_admin_dashboard_stats   subqueries across all core tables
                             → gives: total users, revenue, success rate


================================================================
  SECTION 7 — STORED FUNCTIONS
================================================================

  These are pre-written business logic procedures stored in the DB.
  The backend calls them with SELECT fn_name(params).

  Function                  What it does (ALL steps in one transaction)
  ────────────────────────  ──────────────────────────────────────────────
  fn_confirm_investment     Deduct wallet → create investment → create payment
                            → update project.raised → increment investors_count
                            Returns: 0=OK, 1=no user, 2=no project,
                                     3=low balance, 4=below minimum, -1=error

  fn_top_up_wallet          Add funds → create wallet if needed → write
                            wallet_transaction (credit) → write payment
                            Returns: new balance

  fn_approve_kyc            Set kyc_status='approved', is_verified=TRUE,
                            status='active' (if pending) → write activity log

  fn_reject_kyc             Set kyc_status='rejected' + reason
                            → write activity log

  fn_approve_project        Set project status='active' → write activity log

  fn_reject_project         Set project status='rejected' + reason
                            → write activity log

  fn_refund_payment         Mark payment 'refunded' → cancel investment
                            → reduce project.raised → credit wallet
                            → write wallet_transaction → write activity log

  fn_suspend_user           Set status='suspended' → write activity log
  fn_unsuspend_user         Set status='active'    → write activity log
  fn_ban_user               Set status='banned', freeze wallet → activity log
                            NOTE: permanent — there is no fn_unban_user

  fn_redeem_recharge_card   Validate code → mark redeemed → call fn_top_up_wallet
                            Returns: 0=OK, 1=not found, 2=already used, 3=expired

  Currently both apps run in MOCK MODE (no real backend needed for dev).
  When the backend is built:
    Mobile app  → set useMockApi=false in app.json
    Admin panel → set NEXT_PUBLIC_USE_MOCK=false in .env.local

  The backend will:
    1. Accept the HTTP endpoints listed in backendConfig.js
    2. Map camelCase JSON ↔ snake_case DB columns automatically
       (ASP.NET does this with JsonNamingPolicy.CamelCase)
    3. Query the views and call the functions listed above
    4. Return JSON responses
